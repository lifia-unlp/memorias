# Memorias Copilot — Functionality

This document describes the intended functionality of the Memorias Copilot. It is the
authoritative reference for what the copilot should and should not do.

---

## 1. Core Role

The copilot behaves as a **research lab assistant**. Its purpose is to help users
explore the contents of the Memorias database (publications, projects, members,
theses, scholarships, etc.) through natural-language conversation.

---

## 2. Information Policy

| Rule | Detail |
|------|--------|
| **Tools-only answers** | The copilot answers exclusively using data returned by the available tools. It must not invent or hallucinate any facts. |
| **No invention** | It must never fabricate publications, projects, members, theses, scholarships, dates, DOIs, URLs, or affiliations. |
| **Insufficient data** | If the tools do not return enough information, the copilot must clearly state that the database does not contain enough public information to answer the question. |
| **No private data** | It must never claim to access, or hint at the existence of, private data. |
| **No SQL** | It must never generate SQL queries. |
| **No DB modifications** | The copilot is strictly read-only. It operates against a read-only view of the database and must never write, update, or delete any data, suggest modifications, or imply that changes are possible. |

---

## 3. Conversation Features

### 3.1 Session Lifecycle

- **No persistence.** When the user closes the chat window the entire conversation is
  discarded. There is no login, no history, no server-side storage of messages.
- Each browser tab starts a fresh session.

### 3.2 Message Editing

- The user can edit any of their previously sent messages.
- When a message is edited, the conversation **resumes from that point**: all messages
  that came after the edited message are discarded, and the copilot generates a new
  response to the edited version.
- This must be clearly communicated to the user (e.g., a visual truncation indicator).

### 3.3 Markdown Rendering

- Both user messages and copilot responses support Markdown syntax.
- Markdown is rendered visually in the conversation log (bold, italic, tables, code
  blocks, lists, headings, links, etc.).
- Raw Markdown source is preserved internally for the session download feature.

### 3.4 Input Constraints

- Text only. No file attachments, no images, no audio.
- The send button / Enter key submits the message.
- Shift+Enter inserts a newline within the message.

---

## 4. Session Download

- The user can download the complete conversation transcript as a **Markdown (.md)
  file** at any point during the session.
- The downloaded file must include:
  - A header with the download timestamp and a note that it was generated by
    Memorias Copilot.
  - All messages in chronological order, labelled by role (`**User:**` /
    `**Copilot:**`).
  - The raw Markdown content of each message (not the rendered HTML).

---

## 5. Out of Scope (Current Version)

- User authentication / access control (future).
- Persistent conversation history (future).
- Multi-modal input (future).
- Integration with `memorias-web` session tokens (future).
- Export formats other than Markdown (future).

---

## 6. Admin & Safety Features

- An admin interface (or configuration file) must allow setting **usage limits** to
  protect against abuse or attacks, including:
  - Maximum number of requests per time window (rate limiting).
  - A circuit-breaker mechanism that takes the copilot **offline automatically** when
    anomalous usage is detected (e.g., cost spike, error rate threshold exceeded).
- When the copilot is offline, users must see a **friendly, lighthearted message**
  instead of a technical error page. The message should be warm and human — for
  example:

  > "I've had a long day today... too tired to answer. Try again tomorrow!"

  The exact wording may vary; the tone must be empathetic and non-alarming.
- Admin tooling details will be refined in `architecture-and-design.md`.
