# Memorias Copilot — Project Rules

This document records the rules that govern how this project is built and maintained.
It is the authoritative source for working conventions. Implementation plans and task
lists must not deviate from what is stated here, in `functionality.md`, and in
`architecture-and-design.md`.

---

## 1. Document Authority

- The three project documents (`rules.md`, `functionality.md`,
  `architecture-and-design.md`) are the authoritative source of truth.
- Agent-maintained artefacts (implementation plans, task lists) must stay aligned
  with these documents. Any conflict must be resolved by **first obtaining explicit
  user approval** and then updating the documents accordingly before proceeding.

---

## 2. Source Control

- **Never push to GitHub automatically.** When it is appropriate to commit or push,
  suggest it to the user and wait for their instruction.
- Keep commits atomic and well-described.

---

## 3. Agent Behaviour

- Do not run expensive browser automation or screen-capture tasks unless explicitly
  requested by the user. If that appears to be the only path forward, ask for
  permission first.
- Prefer lightweight, deterministic verification steps (e.g., static analysis, unit
  tests, linting) over interactive browser sessions.
- **Implement automated unit tests** for all backend logic (conversation manager,
  tool dispatcher, database adapter, rate limiter, circuit breaker). Tests must pass
  before any feature is considered done.

---

## 4. Project Isolation

- `memorias-copilot` is a standalone application that must evolve and deploy
  independently from `memorias-web`. Do not introduce cross-project dependencies.
- Before extracting or reusing any component from `memorias-web`, **ask the user**
  whether that is the preferred approach. Do not assume sharing is acceptable.
- Shared utilities, if extraction is approved, must live in a clearly-named shared
  package and the decision must be recorded in `architecture-and-design.md`.

---

## 5. Security & Safety

- The copilot is **strictly read-only**. It operates against a read-only view of
  the database. It must never write, update, or delete any data, directly or
  indirectly.
- The copilot must never access, hint at, or expose private data.
- The copilot must never generate SQL or suggest modifications to the database.
- Admin tooling for usage limits must be in place before any public deployment.
- Access control decisions must be recorded in `architecture-and-design.md`.

---

## 6. Copilot Behaviour Constraints

(These are the hard rules that the copilot's system prompt must enforce; full detail
is in `functionality.md`.)

- Answer only with data returned by the available tools.
- Never invent publications, projects, members, theses, scholarships, dates, DOIs,
  URLs, or affiliations.
- If the tools do not return enough information, say so explicitly.
- Never claim to access private data.
- Never generate SQL.
- Never suggest modifying the database.

---

## 7. Change Management

- Significant architectural or functional changes require updating the relevant
  document(s) and getting user approval before implementation.
- **Before making any large or potentially breaking change**, ensure that all
  outstanding modifications are committed (or stashed) so that a clean rollback is
  possible. Suggest the commit to the user before proceeding.
- Breaking changes to the API layer or database-tool interface must be flagged
  prominently in the PR description (when the user decides to push).

---

## 8. UI & Communication Style

- **No emojis** — neither in the UI (buttons, labels, icons, placeholders) nor in
  any text generated by the copilot. Use SVG icons or plain text instead.
- Keep language professional, concise, and friendly.
